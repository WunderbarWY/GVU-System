# 03-平台适配决策树

本文件列出对现有代码/架构的取舍决策，使用统一符号：

- 🟢 **保留**：现有方案直接使用，无需改造。
- 🟡 **适配改造**：保留核心思路，但需修改实现细节。
- 🔴 **移除/废弃**：不再使用，需清理或替换。
- ⏳ **未来增强**：本次不实现完整功能，但需在代码中预留接口/数据结构。

---

## 1. 本地存储架构

| 项目 | 决策 | 说明 |
|------|------|------|
| `WarHistoryStore` 单例模式 | 🟢 保留 | 继续以对象形式管理内存记录与持久化。 |
| `localStorage` 持久化 | 🟢 保留 | 沿用 `WarHistoryStore.save()` / `load()` 模式。 |
| `G.warHistory` 数组 | 🟡 适配改造 | 保留数组用于兼容旧读取，但写操作必须走 `WarHistoryStore`。 |
| `G.warHistory` 中 `type: 'victory'` | 🔴 移除 | 该类型从未同步，改为统一使用 `WarHistoryStore` 的 `kill`。 |

---

## 2. Supabase 集成

| 项目 | 决策 | 说明 |
|------|------|------|
| `src/api/supabase.js` 封装 | 🟢 保留 | 继续作为 Supabase auth/CRUD 入口。 |
| `_syncRecord` 上传函数 | 🟡 适配改造 | 新增 `client_id` 字段，并实现幂等 upsert。 |
| `war_history` 表 schema | 🟡 适配改造 | 确认/新增 `client_id TEXT UNIQUE` 列。 |
| 云端按 `client_id` 去重 | 🟡 适配改造 | 使用 `upsert` 或唯一约束 + 错误捕获。 |
| 离线同步队列 | ⏳ 未来增强 | 本次实现本地队列 + 恢复同步，云端幂等已为其打基础。 |

---

## 3. Linear 同步逻辑

| 项目 | 决策 | 说明 |
|------|------|------|
| `StarshipSync` 整体架构 | 🟢 保留 | 继续负责 Linear 双向同步。 |
| `StarshipSync.applyIncremental` | 🟡 适配改造 | 检测到 issue 从非 done 变为 done 时，调用 `WarHistoryStore.recordKill`。 |
| `recordSync(sync-out)` 记录 | 🟢 保留 | 仍记录“敌方撤退/同步外出”事件，但不再替代 kill。 |
| Linear done 不再只生成 sync 记录 | 🟡 适配改造 | 补充 kill 记录，而不是替换 sync。 |

---

## 4. 记录函数

| 项目 | 决策 | 说明 |
|------|------|------|
| `recordKill` | 🟢 保留 | 继续作为任务完成记录入口。 |
| `recordDeploy` | 🟢 保留 | 若已存在则保留，若无则无需新增。 |
| `recordPomodoro` | 🟡 适配改造 | 修复 `added`/`removed` 未定义问题；参数改为显式传入对象。 |
| `recordSync` | 🟡 适配改造 | 本地保存后必须调用 `_syncRecord`；支持离线队列。 |

## 5. UI 渲染

| 项目 | 决策 | 说明 |
|------|------|------|
| `renderWarHistory` 情报面板 | 🟢 保留 | 展示最新记录列表。 |

| `type: 'victory'` 的独立展示 | 🔴 移除 | 统一按 `type` 渲染，victory 不再特殊处理。 |
