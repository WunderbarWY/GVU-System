## Phase 0: Schema 更新（10 分钟）

### 目标
确保 `war_history` 表支持 `client_id` 幂等键。

### 任务
1. 检查现有 schema 是否已有 `client_id`。
2. 若无，在 migration 文件或本地 SQL 中新增：
   ```sql
   ALTER TABLE war_history ADD COLUMN IF NOT EXISTS client_id TEXT UNIQUE;
   ```
3. 应用 migration（本地/远程）。

### 通过标准
- `war_history.client_id` 列存在且具有唯一约束。

### 依赖
- Phase -1 完成。

## Phase 1: 修复 recordPomodoro + recordSync（20 分钟）

### 目标
修复两个核心记录函数的 bug。

### 任务
1. 修复 `recordPomodoro` 对未定义变量的引用。
2. 修复 `recordSync` 未调用 `_syncRecord` 的问题。

### 通过标准
- `recordPomodoro({ added: 1, removed: 0 })` 不抛错。
- `recordSync` 调用后触发网络请求或进入队列。

### 依赖
- Phase 0 完成。

## Phase 2: Linear done → recordKill 集成（30 分钟）

### 目标
让 Linear 中标记完成的 issue 正确生成 kill 记录。

### 任务
1. 改造 `StarshipSync.applyIncremental` 检测 done 状态变化。
2. 改造 `WarHistoryStore.recordKill` 支持 `source: 'linear'`。
3. 确保同一 mission 不会同时产生多个 kill。

### 通过标准
- 模拟 Linear done issue 时，`recordKill` 被调用。
- 战争历史记录包含 `metadata.source: 'linear'`。

### 依赖
- Phase 1 完成。

---

## Phase 3: 幂等性 + 离线队列（40 分钟）

### 目标
实现云端去重与离线恢复能力。

### 任务
1. 为 `_syncRecord` 添加 `client_id` 字段与 upsert 逻辑。
2. 在 `src/api/supabase.js` 补充 `upsert` 封装（如缺失）。
3. 新增本地队列（enqueue/dequeue/flushQueue）。
4. 在 `online` 事件与页面加载时调用 `flushQueue`。

### 通过标准
- 重复 upsert 同一 `client_id` 不产生重复行。
- 离线记录保存在 `localStorage`，恢复后同步成功并移除。

### 依赖
- Phase 2 完成。
